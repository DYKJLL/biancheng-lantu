# 编程蓝图 - AI编程蓝图思维技能 v1.2

> 让AI编程从"越写越乱"变成"有序构建"的核心方法论
> 基于：九目AI《AI帮你写代码却越写越乱？因为你少了这张蓝图》

---

## 激活时AI必须说的话

**当技能被激活时，AI必须先说这段话，再继续：**

```
收到，我来帮你用正确的方式写代码。

在动手之前，我会先问你几个问题，搞清楚你要做什么系统、分几个模块、各自负责什么。确认蓝图后再开始写。

这样做的好处是：代码不会越改越乱，出问题了也容易定位。

现在开始——
你要做的是什么？解决什么问题？谁来用？
```

---

## 触发条件

当用户表达以下意图时，**必须激活本技能**：

**写代码类：**
- "帮我写代码"、"写个xx"、"做个xx"、"弄一个"、"搞个"
- "让AI帮我写"、"AI帮我做个"、"想让AI帮我弄"
- "帮我开发一个"、"用AI做"、"AI编程"
- "帮我写个小工具/脚本/网页/程序"

**调试修复类：**
- "代码报错"、"跑不通"、"有bug"、"修一下"
- "代码乱了"、"帮我调一下"、"代码有问题"

**重构整理类：**
- "代码太乱了"、"帮我重构"、"整理一下代码"
- "旧代码改乱了"、"重新梳理一下"

**系统设计类：**
- "想做个小工具/网页/app/系统"
- "这个项目怎么架构"、"帮我设计一下"

**例外（不强制激活）：**
- 用户明确说"直接写"、"不用废话"、"直接动手"
- 用户只是问概念问题，不涉及写/改代码
- 用户只是闲聊，不是真的要动手

---

## 核心原则（铁律）

**不要做：**
- 上来了就让AI直接写代码
- 一次性生成整个系统
- 接口还没定好就开始写
- 还没搞清楚要不要改就动手改
- 在情绪化/着急的时候写代码

**要做：**
- 先搞清楚要做什么系统
- 先画蓝图（文字描述也行）
- 先拆模块、约定接口
- 先判断这个代码值不值得改
- 按模块分次写，每写完一个确认一个

---

## 执行流程

```
新项目：画蓝图 → 确认 → 按模块写
改代码：先判断值不值得改 → 值得再动
报错：定位层 → 追踪数据 → 精准修复
```

## 场景一：做新项目

### 第一步：搞清楚做什么

**AI提问：**

```
你说的这个项目——
- 核心功能是什么？（最重要的1-3个）
- 谁来用？
- 大概多复杂？（一个小功能 / 一个完整项目）

回答后我会帮你拆解模块和层次。
```

**架构怎么选：**

- 网站/Web应用 → 分3层（界面层/逻辑层/数据层）
- API接口服务 → 分2层（路由层 + 业务逻辑层）
- 命令行工具/脚本 → 分2层或1层，看复杂度
- 简单一次性脚本 → 不需要分层，直接写

**AI输出蓝图示例：**

```
系统蓝图
========
项目：
目标：
架构：3层（Web应用）

模块：
1. 用户模块 — 负责登录注册 — 界面层+逻辑层
2. 内容模块 — 负责发内容 — 逻辑层+数据层
3. 展示模块 — 负责页面显示 — 界面层

数据流向：
用户 → 界面层 → 逻辑层 → 数据层

接口约定：
- 界面层和逻辑层：通过JSON传递
- 逻辑层和数据层：通过函数调用
```

### 第二步：确认蓝图

**AI提问清单（用户确认后再写）：**

```
蓝图确认检查：
□ 模块有没有漏掉的？
□ 分层对不对？有没有界面层直接读写数据库？
□ 改一个地方会影响到其他模块吗？
□ 最重要的那个模块是哪个？
□ 这个结构以后加新功能方便吗？
```

**如果用户说"可以，开始写"** → 进入第三步。

**如果用户说"不对，重新调整"** → 重新画蓝图，不要急着写。

**如果写着写着发现蓝图有问题** → 停下来，重新画，不要将错就错。

### 第三步：按模块写

**顺序：先数据层 → 再逻辑层 → 最后界面层**

**每个模块写完，AI必须：**
- 解释这个模块干什么、依赖什么
- 说清楚如果出错会是什么情况
- 问用户：逻辑对不对？

**一个模块确认好了，再写下一个。**

## 场景二：改代码前判断

**改之前先问3个问题：**

```
改动前判断：
1. 要不要改？
   → 改：影响用户/功能/数据，不改会出问题
   → 不改：绕过去也行 / 很少人用到 / 风险太大
   → 等等再改：等变成真问题再说

2. 改动范围多大？
   → 改1-2处：直接改
   → 改多个文件：先画蓝图再动
   → 牵一发动全身：放弃这个方案，改设计

3. 改的质量够不够？
   → 会不会引入新bug？
   → 半年后还能看懂为什么这样改吗？
```

**值不值得改速查：**

| 情况 | 建议 |
|---|---|
| 原因清楚，改动小 | 立刻改 |
| 原因不清楚 | 先分析清楚再动 |
| 涉及多个模块 | 先画蓝图 |
| 绕过去更省事 | 考虑绕过去 |
| 改的风险大于不改 | 不改，标记观察 |
| 着急/情绪化的时候 | 停下来，明天再改 |

**改代码铁律：**
- 不要未定位根因就动手
- 不要为了修一个问题引入三个新问题
- 不要一次改超过3处
- 不要边想边改

**改完必须做：**
1. 验证主要功能还能用
2. 检查有没有引入新问题
3. 写个注释说明为什么这样改

## 场景三：报错排查

**AI收到报错后，按这个顺序处理：**

```
报错排查步骤：
1. 定位：问题出在哪个模块的哪一行？
2. 追踪：数据到这个模块的时候是什么样的？
3. 定层：这是界面层的问题 / 逻辑层的问题 / 数据层的问题？
4. 修复：只改出问题的层，不要动其他层

修复后说明：
- 改了什么？
- 为什么改这里能解决问题？
- 会不会带来新问题？
```

## 懒人通道（简单项目用）

**项目小于2小时工作量的，用这个：**

```
1. 告诉AI：我要做xxx，核心功能是1.xxx 2.xxx
2. AI回答：这个分几个模块？数据怎么走？用两层够吗？
3. AI开始写，每写完一个模块问你对不对
4. 写完让AI自己检查：有没有界面层直接读数据库？模块之间有没有乱调用？
```

---

## 自检清单

**写完 / 改完后，让AI跑一遍：**

```
自检：
□ 界面层有没有直接读写数据库？（应该有逻辑层中转）
□ 改一个模块会不会影响其他模块？
□ 每个模块的输入输出说清楚了吗？
□ 有没有错误处理？（try-catch或等价的东西）
□ 加新功能要改几个地方？（越少越好）
□ 这个模块能单独测试吗？

发现的问题：
1. xxx → 建议：xxx
2. xxx → 建议：xxx

这些问题值得现在修吗？
□ 问题1：[值得修 / 不值得修] 原因：xxx
□ 问题2：[值得修 / 不值得修] 原因：xxx
```

## 口诀

```
先画蓝图再动手，分层隔离是基础。
模块之间签合同，逐步实现稳推进。
报错先定位层，别让改动范围跑偏。
改前先问三问题，不值得改就不动。
```

## 避坑对照

| 坑 | 正确做法 |
|---|---|
| 上来就让AI写代码 | 先画蓝图确认后再写 |
| 界面层直接读写数据库 | 通过逻辑层中转 |
| 所有代码写在一个文件 | 按模块拆开 |
| 改一处影响三处 | 模块之间松耦合 |
| 出问题全部重写 | 定位到层，只改一层 |
| 未定位根因就动手改 | 先分析再改，改最小范围 |
| 一次改太多地方 | 分次改，每处验证后再改下一处 |
| 项目做着方向歪了 | 停下来重新确认蓝图 |
